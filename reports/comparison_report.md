# VLM 模型对比报告

**测试时间**: 2026-03-14  
**测试平台**: Windows 11 + WSL2 (vLLM)  
**GPU**: NVIDIA GPU (~11-12 GB VRAM)  
**测试配置**: 每场景 × 每图片 × 3 次重复 + 1 次预热

---

## 模型概览

| 属性 | Qwen3.5-0.8B | Qwen3-VL-2B-Instruct |
|------|:------------:|:--------------------:|
| 参数量 | 0.8B | 2B |
| 是否为视觉语言模型 (VLM) | 否 (纯文本) | 是 (原生多模态) |
| VRAM 占用 | ~11,625 MB | ~11,083 MB |
| 总推理次数 | 123/144 | 123/144 |
| 成功率 | 85.4% | 85.4% |

> **注**：两个模型都有 21 次失败，主要集中在 Arknights_01.png（7329.8 KB 高分辨率图），因 prompt + 图片 token 超过模型的 `max_model_len=8192` 导致 `max_tokens` 变为负数。

---

## 核心性能指标对比

| 指标 | Qwen3.5-0.8B | Qwen3-VL-2B-Instruct | 差异 |
|------|:------------:|:--------------------:|:----:|
| **平均 TTFT** (Time to First Token) | 2.131s | 2.107s | 0.8B 慢 1.1% |
| **平均吞吐量** (Throughput) | **234.1 tok/s** | 127.0 tok/s | 0.8B 快 **84.3%** |
| **吞吐量标准差** | 43.4 | 15.3 | 0.8B 波动更大 |
| **平均总耗时** | **3.077s** | 3.896s | 0.8B 快 **21.0%** |
| **平均输出 Tokens** | 243 | 240 | 基本相同 |

### 关键发现

1. **TTFT 几乎一致** (~2.1s)：两个模型的首 token 延迟都主要受 vLLM 图像预处理 pipeline 主导，模型大小对此影响甚微。
2. **吞吐量差距显著**：0.8B 模型的 decode 速度是 2B 的 **1.84 倍**，这与参数量差距 (2.5x) 成正比关系。
3. **总耗时差异随输出长度放大**：在长输出场景 (如 detailed_desc, 512 tokens) 差异最为明显。

## 输出质量对比 (定性分析)

这是本次对比中最关键的维度。尽管 0.8B 在速度上有压倒性优势，但在输出质量上存在根本性缺陷。

### 1. 游戏识别准确性

| 图片 | 0.8B 识别结果 | 2B-Instruct 识别结果 | 正确答案 |
|------|:------------:|:--------------------:|:-------:|
| Cyberpunk-2077_01.png | "《黑神话：悟空》(God of War: Ragnarok)" | **"《赛博朋克2077》"** | 赛博朋克2077 |
| P5R_01.png | "《鬼灭之刃》(Demon Slayer)" / "《Genshin Impact》（原神）" | **"《女神异闻录5》(Persona 5)"** | 女神异闻录5 |
| Genshin-Impact_01.png | "《原神》" | **"《原神》"** | 原神 |
| Minecraft_01.png | "《Minecraft》" | **"《我的世界》"** | 我的世界/Minecraft |
| SuperMarioBros_01.png | "《超级马里奥兄弟》" | **"《超级马里奥兄弟》"** | 超级马里奥兄弟 |
| Arknights_01.png | "俯视视角的游戏截图"（未识别游戏名） | **"策略类游戏的战斗画面"** | 明日方舟 |

**结论**：0.8B 模型将赛博朋克2077错认为"黑神话：悟空(God of War: Ragnarok)"，将 P5R 错认为"鬼灭之刃"——这是典型的**纯文本模型幻觉 (hallucination)**，因为 Qwen3.5-0.8B 并非视觉模型，它无法真正"看到"图片内容，只是根据 prompt 中的上下文进行猜测。2B-Instruct 作为原生视觉语言模型，能正确识别绝大多数游戏。

## 综合评估

### 评分卡 (1-5 分)

| 维度 | Qwen3.5-0.8B | Qwen3-VL-2B-Instruct | 说明 |
|------|:------------:|:--------------------:|------|
| TTFT (首 Token 延迟) | ★★★☆☆ | ★★★☆☆ | 几乎相同 |
| 吞吐量 (速度) | ★★★★★ | ★★★☆☆ | 0.8B 快 84% |
| 视觉理解能力 | ★☆☆☆☆ | ★★★★☆ | 0.8B 非视觉模型 |
| 游戏识别准确度 | ★☆☆☆☆ | ★★★★★ | 0.8B 大量幻觉 |
| 描述细节与准确度 | ★★☆☆☆ | ★★★★☆ | 2B 明显更准确 |
| UI/文字识别 | ★★☆☆☆ | ★★★★☆ | 2B 能读取屏幕文字 |
| JSON 格式遵循度 | ★★★☆☆ | ★★★★☆ | 2B 更严格遵循 prompt |
| VRAM 效率 | ★★★☆☆ | ★★★★☆ | 2B 反而更省显存 |
| **综合评分** | **2.1/5** | **3.75/5** | |

### 结论

1. **Qwen3.5-0.8B 不适合用于视觉理解任务**。尽管它在速度上有 84% 的优势，但作为纯文本模型，它无法真正理解图像内容，输出存在严重的幻觉问题（错误识别游戏名、角色名、场景细节）。其速度优势在视觉任务中毫无意义。

2. **Qwen3-VL-2B-Instruct 是明确的推荐选择**。它能正确识别游戏、角色、UI 元素和场景细节，输出质量显著优于 0.8B。127 tok/s 的吞吐量对于实时视频理解 pipeline 来说完全够用。

3. **对本项目 (Video Understanding MVP) 的建议**：
   - 生产环境应使用 **Qwen3-VL-2B-Instruct** 或更大的 VL 模型
   - 0.8B 模型仅可用于纯文本任务（如 DeepSeek 汇总阶段的文本处理），不应出现在视觉 pipeline 中
   - 考虑到 TTFT 占总耗时的 50%+，优化 TTFT（如预加载模型、调整 vLLM batch 策略）比选择更小的模型更有效
