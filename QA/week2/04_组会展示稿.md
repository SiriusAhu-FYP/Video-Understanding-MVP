## 系统架构：视频/窗口理解流水线

本周实现的系统原型（`pipeline/` + `experiments/video_understanding/`）

```
┌─────────────────────────────────────────────────────┐
│                    Windows 宿主机                     │
│                                                       │
│  PotPlayer / 游戏窗口                                  │
│       │                                               │
│       ▼  每 500ms (win32gui + mss)                    │
│  截图捕获 capture_window()                             │
│       │                                               │
│       ▼  MSE 帧差 compute_diff()                      │
│  关键帧过滤  (阈值 500, 丢弃静止帧)                    │
│       │                                               │
│       ▼  Base64 编码                                  │
│  KeyFrameQueue (异步队列, 超时自动丢弃)               │
│       │                                               │
│       ▼  HTTP (OpenAI 协议)                           │
│  ┌────────────────────────────────┐                  │
│  │   WSL2 vLLM 服务 (port 8000)   │                  │
│  │   VLMClient.describe_frame()   │                  │
│  └────────────────────────────────┘                  │
│       │                                               │
│       ▼  帧描述文本 (异步累积)                         │
│  Summarizer (DeepSeek API)                            │
│       │                                               │
│       ▼                                               │
│  summary.txt / report.md                             │
└─────────────────────────────────────────────────────┘
```

## 系统核心模块说明

| 模块 | 文件 | 功能 |
|------|------|------|
| 窗口捕获 | `pipeline/capture.py` | win32gui 枚举窗口 + mss 极速截图 |
| 帧差过滤 | `pipeline/capture.py` | MSE / SSIM 计算，过滤无变化帧 |
| 异步队列 | `pipeline/queue_manager.py` | 生产者-消费者解耦，超时自动丢弃 |
| VLM 客户端 | `pipeline/vlm.py` | OpenAI 兼容协议，支持视觉输入 |
| 汇总器 | `pipeline/summarizer.py` | 调用 DeepSeek API 生成整体摘要 |
| 实验编排 | `experiments/video_understanding/run_experiment.py` | 流水线控制 + 播放器管理 |

**关键设计**：截图（生产者）和 VLM 推理（消费者）完全异步解耦。截图不等推理完成，推理不阻塞截图。若推理来不及，超时帧自动丢弃而非积压。

## 实验设计（续）

**实验一：推理速度基准（benchmark_speed）**

- 5 种场景 × 6 张游戏截图 × 3 次重复 = **每模型 90 次推理**
- 测量指标：首 Token 延迟（TTFT）、吞吐量（tokens/s）

**实验二：视频理解流水线（video_understanding）**

```
PotPlayer 播放录屏
    → 每 500ms 截图（2 FPS）
    → MSE 帧差过滤（阈值 500）
    → 关键帧入队
    → VLM 异步推理
    → DeepSeek 汇总
```

- 2 个游戏视频（原神 24.5s、MC 20s）× 3 次重复 = **每模型 6 次运行**

**硬件环境**：RTX 4080 SUPER (16GB) · vLLM 0.17.0 · WSL2

## 结果：视频理解实时性

**帧丢弃率** = 关键帧因推理积压超时被丢弃的比例（越低越好）

| 模型 | 原神视频丢弃率 | MC 视频丢弃率 |
|------|:---:|:---:|
| DS-VL2-tiny | **~17%** | **~5%** |
| InternVL2.5-2B | ~31% | ~25% |
| Qwen3-VL-2B | ~29% | ~23% |
| Qwen3.5-0.8B | ~43% | ~45% |
| SmolVLM2-2.2B | ~39% | ~67% |
| Phi-3.5-vision | ~80% | ~79% |

> ⚠️ **SmolVLM2 速度最快，但丢弃率并不最低！**

---

## 关键发现 1：速度 ≠ 实时性

**SmolVLM2 的悖论**：

- 吞吐量最高（114.9 tok/s）✅
- 但 MC 视频丢弃率高达 **67%** ❌

**原因**：SmolVLM2 的描述偏泛化，每次推理产生的有效信息少，系统整体效率低。

**结论**：评估实时系统不能只看吞吐量，需要看**端到端的帧处理率**。

---

## 关键发现 2：参数量 ≠ 性能

```
4.2B Phi-3.5  →  49.2 tok/s（排名第 6）
2.2B SmolVLM2 →  114.9 tok/s（排名第 1）
```

**4.2B 的模型比 2.2B 的模型慢了 2.3 倍**

原因：架构设计和 vLLM 框架优化程度才是决定性因素，而非参数量。

---

## 关键发现 3：Mamba 架构在当前框架下表现不佳

**Qwen3.5 系列（Mamba 混合架构）**：

```
Qwen3.5-2B  (2B 参数):  47.2 tok/s
Qwen3.5-0.8B (0.8B 参数): 45.7 tok/s
```

参数量减少 **60%**，速度只提升 **3%**（统计上无意义）

**原因分析**：
1. vLLM 0.17.0 对 Mamba 的 CUDA kernel 优化不成熟
2. Mamba 的优势在长序列（>2K tokens），我们的任务是短序列（64-512 tokens）
3. Mamba 需要更多 VRAM（gpu-memory-utilization 需要 0.7，而非 0.5）

## 遇到的问题

| 问题 | 原因 | 解决方案 |
|------|------|------|
| MARTHA-2B / moondream2 无法加载 | 架构不兼容 vLLM | 替换为 Phi-3.5 / SmolVLM2 |
| Mamba 模型启动失败 | KV Cache 不足（0.5 显存利用率） | 提升至 0.7 |
| PotPlayer 不自动关闭 | WM_CLOSE 消息无响应 | psutil 强制终止 + try/finally |
| 帧差计算报错 | 截图分辨率不一致 | cv2.resize 容错 |
| SmolVLM2 启动失败 | 缺少 num2words 依赖 | pip install num2words |

## Q&A 速查：高频问题一句话答案

> 组会前 5 分钟快速扫一遍这张表。

| 问题 | 一句话答案 |
|------|------|
| TTFT 是什么？ | Time to First Token，从发请求到收到第一个 token 的时间，反映 prefill 速度 |
| 帧丢弃率是什么？ | 关键帧在队列中超时没被消费，自动丢弃的比例；越低说明模型跟得上截图频率 |
| 为什么 SmolVLM2 最快但丢弃率高？ | 描述泛化、输出 token 少，总推理时间和 InternVL2.5 差不多，但在 MC 慢节奏视频里积压更严重 |
| 为什么 Mamba 不快？ | vLLM 0.17.0 对 Mamba kernel 优化不成熟；Mamba 优势在长序列（>2K），我们任务是短序列（64-512） |
| 为什么不用 GPT-4V？ | 云端 API，延迟 1-5s，有成本；我们需要本地部署、零边际成本 |
| MSE 阈值 500 怎么定的？ | 经验调整：在原神视频上手动观察，该阈值过滤 ~17% 静止帧同时不丢失场景切换 |
| 系统能处理游戏以外的窗口吗？ | 能，只需更换窗口关键词；系统是通用的窗口截图理解工具 |
| 为什么要用 WSL2？ | vLLM 不支持 Windows，必须在 Linux 环境（WSL2）运行；截图捕获依赖 Windows API |
| 下周做什么？ | 补充描述质量人工评分 + INT4 量化实验 + 真实游戏（非录屏）测试 |
| FYP 最终目标是什么？ | 游戏智能陪玩系统：实时理解画面 → 给出策略建议/解说 |
