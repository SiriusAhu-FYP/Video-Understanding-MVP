# SOTA 模型验证实验设计文档

## 1. 目标

本实验的目标不是比较不同模型的商业价值，而是**验证当前实验设计本身是否合理**。

原则如下：

1. 使用一组当前可获得的顶级多模态 / 视觉模型，对既定的图片与视频样本进行统一评测。
2. 如果这些 SOTA 模型在我们的样本上都能得到接近满分，说明：

   * 样本本身信息充分；
   * 提示词设计合理；
   * 评分标准不过分苛刻；
   * 该实验可以继续用于后续小模型对比。
3. 如果多个 SOTA 模型在某个样本上普遍分数偏低，则优先怀疑：

   * 样本设计有歧义；
   * 提示词不清楚；
   * 评分 rubric 过严或不稳定；
   * 视频处理流程存在问题。

本实验属于 **quality 校准实验**，用于验证实验设计，而不是生成正式排行榜。

## 2. 模型选择原则

### 2.1 选择标准

纳入验证组的模型应尽量满足以下条件：

* 当前公认前沿或接近前沿；
* 明确具备视觉理解能力；
* 至少适合图像理解；
* 如用于视频实验，则应具备原生视频理解能力，或在拆帧后具备稳定的时序总结能力；
* 来自不同技术路线与厂商，避免单一路线偏差。

### 2.2 本次使用的模型

本次视觉验证实验使用以下模型：

```text

Qwen3-VL-235B-A22B-Instruct
qwen3.5-397b-a17b
qwen2.5-vl-72b-instruct
gpt-4.1
claude-sonnet-4-6
grok-4.1
glm-4.6v
```

### 2.3 模型选择理由

* **gemini-3.1-pro-preview**：Google 官方明确定位为高级多模态模型，支持图像和视频，适合作为图像与视频共同的强基线。 ([Google AI for Developers][4])
* **gpt-5.4**：OpenAI 官方文档明确支持图像输入，并且定位为 frontier model，适合做高质量图像理解校准。 ([OpenAI Developers][3])
* **claude-opus-4-6**：Anthropic 官方建议用于最复杂任务；Claude 官方文档明确支持 vision，适合做高质量图像样本校准。 ([Anthropic][5])
* **Qwen3-VL-235B-A22B-Instruct**：Qwen 官方明确将其定位为文本、图像、视频统一理解的大模型，是开源/开放权重阵营中很重要的视觉强基线。 ([Hugging Face][6])
* **qwen3.5-397b-a17b**：Qwen3.5 官方页面明确强调 unified vision-language foundation，并声称在视觉理解基准上强于 Qwen3-VL 系，是值得纳入的最新路线。 ([Hugging Face][7])
* **qwen2.5-vl-72b-instruct**：Qwen 官方明确强调其视频理解、长视频理解与视觉定位能力，适合作为视频强基线之一。 ([Hugging Face][8])
* **gpt-4.1**：OpenAI 官方明确支持图像输入，适合作为高稳定性的图像理解参照模型。 ([OpenAI Developers][9])
* **claude-sonnet-4-6**：Claude Sonnet 4.6 是 Anthropic 当前更轻量但仍很强的一代模型，支持 vision，适合作为 Opus 的补充对照。 ([Anthropic][10])
* **grok-4.1**：xAI 官方材料明确指出 Grok 4 / 4.1 路线具备 multimodal / visual processing 能力，可作为另一家前沿路线的补充。 ([xAI][11])
* **glm-4.6v**：Z.AI 官方的 glm-4.6v 明确支持图像输入和图像分析。

## 3. API 与环境配置

### 3.1 API 端点

统一使用 DMXAPI：

```text
https://www.dmxapi.cn/v1
```

### 3.2 `.env`

在 `.env` 中添加与 DMXAPI 相关配置。
代码需从环境变量中读取，不允许硬编码密钥。

建议字段：

```env
DMXAPI_BASE_URL=https://www.dmxapi.cn/v1
DMXAPI_API_KEY=your_api_key_here
DMXAPI_TIMEOUT_SECONDS=180
DMXAPI_MAX_RETRIES=3
DMXAPI_RETRY_WAIT_SECONDS=60
```

### 3.3 `.env_example`

必须补充 `.env_example`，内容应包含但不暴露真实密钥：


## 5. 实验原则

### 5.1 所有提示词必须保持一致

这是硬要求：

* **SOTA 校准实验与小模型实验使用相同的主提示词模板**
* 不允许因为模型更强就修改任务提示词
* 视频实验中，**负责总结的模型可以是被测模型自己**，但前置输入与任务定义必须与正式实验保持一致

目的：保证比较的是模型能力，而不是提示词优化后的能力。

### 5.2 每个图片 / 视频都做 5 次实验

每个模型对每个样本都运行 **5 次**，用于观测稳定性。建议也通过一个单独的 toml 文件来配置，不要和主实验的 toml 文件混在一起。

目标不是只看单次最好结果，而是看：

* 平均分
* 方差 / 波动
* 是否存在明显抽风

## 6. 预检（必须先做）

在正式批量实验前，必须先做一个**低成本连通性预检**。

### 6.1 预检要求

对每一个待测模型：

1. 先用一张**最短边不低于 512px 的真实游戏截图**（从 `assets/images/` 中取第一张并等比缩放）发起一次最简请求；
2. 验证该模型：

   * 能正常鉴权；
   * 能正常返回；
   * 不会立刻报参数错误 / 模型不可用 / 多模态不支持；
3. 只有预检通过，才允许进入正式实验。

> **注意**：预热图片不应太小（如 64×64 纯色块），否则模型看不清任何内容，容易返回空响应导致误判为不可用。最短边至少 512px 即可保证模型有足够信息进行判断。

### 6.2 预检目的

避免：

* 模型名写错；
* 模型根本不支持图像；
* 请求格式不兼容；
* 批量跑到一半才发现浪费大量 token。

### 6.3 预检并行

所有模型的预检可以**并行执行**（`asyncio.gather`），无需逐个串行，以加速预热阶段。

## 7. 限流与重试

实验过程中，必须识别类似“**一分钟内请求次数过多**”这类限流错误。

### 7.1 限流处理要求

如果返回内容中可识别出：

* rate limit
* too many requests
* requests per minute
* retry after
* 一分钟内请求次数过多

则应：

1. 记录该错误；
2. 自动等待 **60 秒**；
3. 再次重试；
4. 保持实验流程连续，避免中断导致整批结果碎裂。

### 7.2 普通错误处理

普通错误应区分：

* 网络错误
* 模型不可用
* 请求格式错误
* 内容过大
* 超时

并在结果日志中记录。

## 8. 视频处理方式

### 8.1 原则

视频实验不应每次都重新实时截图，否则会非常慢、非常贵、非常不稳定。

### 8.2 正确流程

对每个视频样本：

1. **只在最开始播放或读取一次视频**
2. 从该视频中预先抽取固定图片序列
3. 将这组图片序列缓存到本地
4. 后续所有模型、所有轮次都复用同一组帧
5. 再把这些帧统一发送给云端模型进行解析
6. 最终让模型基于同一帧序列完成总结

### 8.3 好处

这样可以保证：

* 每个模型看到的是完全相同的视频信息；
* 不会因为重复截帧导致输入不一致；
* 成本更可控；
* 结果更可复现。

## 9. 评分方法

### 9.1 评分核心

实验流程分两层：

1. **被测 SOTA 模型** 先完成任务回答；
2. 再由 **DeepSeek（`deepseek-chat`）统一作为裁判模型**，根据样本 JSON 中的 `grading_prompt_for_judge_model` 和 `reference_answer` 进行打分。

> **设计决策**：打分模型统一使用 DeepSeek，而不是让被测模型自己打分。原因：
> * 避免不同模型对评分 rubric 的理解差异导致分数不可比；
> * DeepSeek 作为纯文本模型打分，成本远低于 SOTA 视觉模型；
> * 与现有 `benchmark_quality` 实验保持一致（该实验同样使用 DeepSeek 作为 judge）。

### 9.2 评分目标

本阶段不是追求模型绝对排名，而是验证样本质量。

因此更关注：

* 每个样本的平均分是否足够高
* 多个顶级模型是否都能稳定高分
* 是否存在某个样本让大部分顶级模型失分严重

### 9.3 建议判定规则

一个样本可以初步视为“设计合理”，当满足：

* 多数 SOTA 模型平均分较高；
* 没有大量低分异常；
* 失分主要集中在非核心细节；
* 不会普遍在 must-hit points 上失败。

如果多个顶尖模型在同一样本上都明显翻车，则应优先怀疑样本设计或评分规则，而不是认为这些模型都“不够强”。

---

## 9.5 并行执行

### 9.5.1 并行策略

所有模型的实验（图片和视频）均可**并行执行**：

* 使用 `asyncio.gather` 同时发起所有模型的实验任务；
* 实际上 DMXAPI 不存在严格的"每分钟请求数"限制，只需在收到错误响应后做好重试即可；
* 每个模型的结果写入各自独立的目录（`output_dir/{model_name}/results.csv`），因此并行写入不会冲突。

### 9.5.2 错误处理

* 任何单个模型的 API 错误不应影响其他模型的并行执行；
* 每个请求失败后独立重试，不影响其他请求的记录；
* 最终收集所有模型的结果后统一聚合。

---

## 9.6 可视化

实验完成后，自动生成以下图表（保存到 `session_dir/charts/`）：

1. **总分排名**（水平条形图）：按模型的跨样本平均分排名；
2. **维度热力图**（Heatmap）：模型 × 评分维度的平均分矩阵；
3. **稳定性对比**（条形图）：每个模型的标准差，越低越稳定；
4. **样本对比**（分组条形图）：每个样本在各模型上的平均分，便于发现问题样本；
5. **延迟对比**（条形图）：每个模型的平均 VLM 响应延迟。

图表风格与主实验（`ahu_paimon_toolkit.utils.visualization`）保持一致。

## 10. 输出结果

### 10.1 原始输出

每一次实验都应保存：

* 模型名
* 样本 id
* 轮次编号（1~5）
* 原始回答
* 裁判评分结果
* 失败 / 重试记录
* 时间戳
* 耗时

### 10.2 聚合结果

对每个模型 × 样本，至少聚合：

* 平均分
* 每维度平均分
* 最低分
* 最高分
* 标准差或波动描述
* 失败次数
* 重试次数

## 11. QA 报告

运行完实验后，在 `v3` 下单独创建一个文件夹保存 QA 报告。

建议结构：

```text
v3/
└── sota_validation/
    └── 评估报告.md
```

### 11.1 `评估报告.md` 应说明

1. 本实验是如何做的；
2. 使用了哪些模型；
3. 如何做预检；
4. 如何处理限流与重试；
5. 视频是如何抽帧与复用的；
6. 每个图片 / 视频共做了多少次实验；
7. 结果总体如何；
8. 哪些样本是稳定高分；
9. 哪些样本存在争议或需修改。

至少回答：

* 哪些 SOTA 模型能够稳定高分？
* 哪些样本对 SOTA 模型仍不稳定？
* 图片与视频样本中，哪个最容易造成分歧？
* 当前 4 图 + 2 视频是否适合作为下一步小模型实验基准？
* 如果不适合，应改哪些样本或改哪里？
