# 多模型视觉语言模型综合基准测试 - 实验设计 v3

> 本文件基于 `综合实验设计_v2.md`，结合 `QA/week2/02_实验设计自省.md` 中指出的问题进行重大升级。
> 核心改进：引入 **LLM-as-Judge 质量评估**、**任务驱动提示词**、和 **更严格的统计实践**。

## 一、实验目的

对多款小型视觉语言模型（VLM）进行 **推理速度**、**描述质量** 和 **视频理解能力** 的全面量化评估，服务于**游戏画面实时理解**场景。

### v2 → v3 核心改进

| 维度 | v2 | v3 |
|------|----|----|
| 质量评估 | 仅主观 | LLM-as-Judge 量化评分（0-10 分制） |
| 提示词 | 通用 `prompts/` 文件 | JSON 文件内嵌任务专属提示词 |
| 评估场景 | 2 个实验 | 3 个实验（新增 benchmark_quality） |
| 重复次数 | 3 | 5（配 95% CI） |
| 统计指标 | mean ± std | mean ± std + 95% CI + 维度分解 |
| 工具包 | `toolkit/` | `ahu_paimon_toolkit` (解耦子模块) |

## 三、实验内容

### 3.1 benchmark_speed（推理速度基准）

**目的**：测量各模型在不同任务场景下的推理延迟和吞吐量。

- **场景数量**：5 个
  - `short_desc`：短文本描述
  - `detailed_desc`：详细描述
  - `object_detection`：目标检测
  - `action_analysis`：动作分析
  - `ui_recognition`：UI 识别
- **图片数量**：4 张（来自 `assets/images/`）
- **重复次数**：5 次 + 1 次预热（v2: 3 次）
- **每模型总推理次数**：5 × 4 × 5 = 100 次
- **指标**：TTFT (s)、Generation Throughput (tokens/s)、GPU VRAM (MB)

**v3 改动**：提示词现在使用 JSON 文件中的 `prompts.A_description` 而非硬编码字符串。重复次数从 3 提升至 5。

### 3.2 benchmark_quality（描述质量基准）🆕

**目的**：量化评估各模型对游戏画面的理解和描述质量。**这是 v3 最重要的新增实验。**

#### 流程

```
for each image_asset in assets/images/:
    load JSON task definition
    for prompt_mode in [A_description, B_assistant]:
        for run in range(5):
            response = vlm_client.chat(prompt_mode, image)
            score = judge.evaluate(response, reference, rubric)
            save(score)
```

#### LLM-as-Judge 评分机制

- **评审模型**：DeepSeek (`deepseek-chat`)
- **评分维度**（每项 0-2 分，共 5 维度，满分 10）：
  - 图片任务：`core_understanding`, `key_information_coverage`, `task_completion`, `assistant_value`, `hallucination_control`
  - 视频任务：`event_sequence_understanding`, `key_change_recognition`, `outcome_understanding`, `assistant_value`, `hallucination_control`
- **评分依据**：
  - 参考答案（minimum_expected_points + good_additional_points）
  - 评分标准（每维度的 0/1/2 判定准则）
  - 硬性扣分项（hard_penalties）
- **输出格式**（JSON）：

```json
{
    "dimension_scores": {"core_understanding": 2, "key_information_coverage": 1, ...},
    "total_score": 7,
    "strengths": ["..."],
    "weaknesses": ["..."],
    "missing_points": ["..."],
    "hallucinations": ["..."]
}
```

#### 统计指标

- 每模型每素材：5 次重复 × 2 prompt_mode = 10 次评分
- 汇总：mean ± stdev + 95% CI
- 维度分解：per-dimension mean across all assets

### 3.3 video_understanding（视频理解）

**目的**：测试完整实时流水线（截图 → 帧差 → VLM → 汇总）的端到端效果。

- **视频素材**：2 个游戏录屏
- **采样间隔**：500ms
- **帧差算法**：MSE (阈值 500.0)
- **重复次数**：每视频 5 次（v2: 3 次）
- **每模型总运行次数**：2 × 5 = 10 次完整流水线
- **质量评估**：每次运行的最终总结也送入 LLM-as-Judge 评分

**v3 改动**：增加质量评分环节。流水线运行后将 DeepSeek 总结文本送入 Judge 进行评估，使用视频素材 JSON 中的 rubric。

## 六、实验编排

### 6.1 整体流程

```
for model in models:
    1. kill_all_vllm()
    2. start_vllm(model)
    3. wait_for_ready(300s)
    4. run_benchmark_speed()        # 100 次推理
    5. run_benchmark_quality()      # 40 次推理 + 40 次 Judge 评分 🆕
    6. run_video_understanding()    # 10 次流水线 + 质量评分 🆕
    7. stop_vllm()

generate_comparison_reports()       # 速度 + 质量 + 视频 综合对比
generate_quality_heatmaps()         # 维度得分热力图 🆕
```

### 6.2 关键工程细节

1. **Base URL 检测**：通过 `get_wsl_ip()` 获取 WSL2 的 eth0 IP。
2. **播放器进程管理**：WM_CLOSE → wait(3s) → kill → kill_all (psutil)。
3. **帧差计算容错**：`compute_diff()` 在 shape 不匹配时自动 resize。
4. **vLLM 进程清理**：`pkill -9 -f vllm` + `fuser -k 8000/tcp`。
5. **Judge 调用限频**：DeepSeek API 有 QPM 限制，在评分调用间插入适当间隔。

### 6.3 benchmark_quality 细节

```python
for asset_json in load_all_asset_jsons("assets/images/"):
    image_b64 = encode_image(asset_json["image_file"])
    for prompt_mode in ["A_description", "B_assistant"]:
```

## 八、统计方法（v3 新增）

### 8.1 评分可靠性

- 每个条件（模型 × 素材 × prompt_mode）重复 5 次
- 报告 mean ± stdev 和 95% CI

### 8.2 置信区间计算

```
CI_95 = mean ± 1.96 × (stdev / √n)
```

其中 n = 重复次数 (5)。

### 8.3 多维度分析

- 每个维度单独报告 mean score
- 热力图展示 模型 × 维度 的得分矩阵
- 识别各模型的「强项」和「弱项」维度

## 九、研究问题

沿用 v2 的三个研究问题，新增以下：

4. **描述质量排名**：哪个模型在游戏画面理解的质量评分上表现最好？质量与速度之间是否存在权衡？
5. **维度差异**：不同模型在 hallucination_control、core_understanding 等维度上是否有显著差异？
6. **提示词敏感性**：模型对 A_description vs B_assistant 两种提示词的响应质量是否有显著差异？

## 十、局限性

1. LLM-as-Judge（DeepSeek）本身可能存在评分偏差，非人类标注。
2. 素材数量（4 图 + 2 视频）仍然有限，但已通过结构化 rubric 提升评估深度。
3. 仅测试 FP16/BF16 权重，未包含量化模型。
4. 实验环境依赖 WSL2 + Windows，可移植性有限。
