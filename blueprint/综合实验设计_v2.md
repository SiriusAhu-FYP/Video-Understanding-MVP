# 多模型视觉语言模型综合基准测试 - 实验设计 v2

> 本文件基于 `综合实验设计.md` (v1)，根据 2026-03-15 ~ 2026-03-16 的实际实验经验进行修正和补充。

## 一、实验目的

对多款小型视觉语言模型（VLM）进行推理速度和视频理解能力的全面评估，服务于**游戏画面实时理解**场景。

## 二、准备工作

与 v1 基本一致，但有以下修正：

1. ~~`gameplay_analysis` 放在 toolkit 下~~ → `gameplay_analysis` 保留在 `experiments/` 下作为独立实验，`toolkit/` 仅存放通用工具（`common.py`, `visualization.py`, `vllm_manager.py`）。
2. 综合测试入口脚本：`run_benchmark.py`（而非 v1 中的 `run_multi_model_benchmark.py`）。
3. 配置文件：`config_benchmark.toml`（而非 `config_multi_model_benchmark.toml`）。
4. **新增依赖**：`psutil`（播放器进程管理）。WSL 环境需要 `num2words`（SmolVLM2 专用）。

## 三、实验内容

### 3.1 benchmark_speed（推理速度基准）
- **场景数量**：5 个（v1 原为 8 个，v2 移除了场景分类、中英双语描述和叙事性描述）
  - `short_desc`：短文本描述
  - `detailed_desc`：详细描述
  - `object_detection`：目标检测
  - `action_analysis`：动作分析
  - `ui_recognition`：UI 识别
- **图片数量**：6 张游戏截图
- **重复次数**：3 次 + 1 次预热
- **每模型总推理次数**：5 × 6 × 3 = 90 次

### 3.2 video_understanding（视频理解）
- **视频素材**：2 个游戏录屏（原神 24.5s、MC 20s）
- **采样间隔**：500ms
- **帧差算法**：MSE (阈值 500.0)
- **重复次数**：每视频 3 次
- **每模型总运行次数**：2 × 3 = 6 次完整流水线

## 五、vLLM 服务配置（v2 修正）

### 5.1 默认参数

```bash
vllm serve "<model_id>" \
    --trust-remote-code \
    --max-model-len 4096 \       # v1 为 8192，减小以降低 KV Cache 内存压力
    --gpu-memory-utilization 0.7 \ # v1 为 0.5，提高以支持 Mamba 模型
    --enforce-eager               # v2 新增，禁用 CUDA Graph 以加快启动和减少显存开销
```

### 5.2 关键修正

1. **`gpu-memory-utilization` 从 0.5 提升至 0.7**：Mamba 混合架构（Qwen3.5 系列）在 0.5 下因 KV Cache 不足触发 `AssertionError: num_cache_lines >= batch`。0.7 可满足所有 8 个模型。

2. **`max-model-len` 从 8192 降至 4096**：减小可分配的最大序列长度，降低 KV Cache 内存需求，使更多显存可用于模型权重和激活值。

3. **新增 `enforce-eager`**：禁用 CUDA Graph 编译。CUDA Graph 虽可加速推理，但编译过程需要额外显存和时间（3-5 分钟）。对于单 GPU 小模型评测场景，enforce-eager 更实用。

4. **移除 `enable-prefix-caching`**：v1 启用了 prefix caching，但在单轮推理评测中无效（无共享前缀）。

### 5.3 模型特殊参数

| 模型 | 额外参数 |
|------|------|
| mistralai/Ministral-3-3B | `--tokenizer_mode mistral --config_format mistral --load_format mistral` |
| deepseek-ai/deepseek-vl2-tiny | `--hf-overrides '{"architectures":["DeepseekVLV2ForCausalLM"]}'` |
| 其他模型 | 无额外参数 |

### 5.4 环境依赖

- vLLM 虚拟环境需安装 `timm`（DeepSeek-VL2 依赖）。
- vLLM 虚拟环境需安装 `num2words`（SmolVLM2 依赖）。
- WSL 中使用 `HF_HUB_OFFLINE=1 TRANSFORMERS_OFFLINE=1` 避免网络问题。

## 六、实验编排

### 6.1 流程

```
for model in models:
    1. kill_all_vllm()       # 清理残留进程
    2. start_vllm(model)     # 启动 vLLM 服务
    3. wait_for_ready(300s)  # 轮询 /v1/models 直到就绪
    4. run_benchmark_speed() # 90 次推理
    5. run_video_understanding() # 6 次流水线
    6. stop_vllm()           # 停止服务
```

### 6.2 关键工程细节

1. **Base URL 检测**：通过 `get_wsl_ip()` 获取 WSL2 的 eth0 IP（而非 localhost），因为 vLLM 运行在 WSL 而 Python 脚本运行在 Windows。

2. **播放器进程管理**（v2 新增）：
   - `launch_player()` 返回 `subprocess.Popen` 句柄
   - `close_player()` 实现「先礼后兵」策略：WM_CLOSE → wait(3s) → kill → kill_all
   - `_kill_all_players()` 使用 `psutil` 强制终止所有 PotPlayer 进程
   - `try/finally` 包裹流水线，确保异常时也清理播放器

3. **帧差计算容错**（v2 新增）：`compute_diff()` 在 frame_a.shape ≠ frame_b.shape 时自动 resize，避免 PotPlayer 窗口尺寸变化导致的 ValueError。

4. **vLLM 进程清理**：使用 `pkill -9 -f vllm` + `fuser -k 8000/tcp` 双重保障，每次启动新模型前彻底清理。

## 八、研究问题

1. **理论最快帧率**：TTFT 的倒数即为理论最大 FPS。实测结果显示所有模型的 TTFT 在 0.11-0.20s，理论 5-9 FPS。
2. **安全帧率**：取理论最大 FPS 的 70% 作为安全值（预留网络、队列、帧差计算开销）。实际采样 2 FPS 即可满足。
3. **未处理帧的命运**：超出队列容量的帧被丢弃（`dropped_count`），过期帧也会自动清除。不会造成「信息一直滞后」的连锁反应，但高丢弃率意味着实时覆盖度下降。

## 九、关于量化模型的补充说明

本实验仅测试了 HuggingFace 官方发布的 FP16/BF16 权重。社区制作的量化版本（如 `unsloth` 的 GGUF、AWQ、GPTQ 等）可能在推理速度和显存占用上有进一步优化，但：

1. vLLM 对 GGUF 的支持有限（部分架构不兼容）。
2. 量化可能影响视觉理解质量，需要单独的质量评估。
3. 建议在后续实验中选择 1-2 个表现最优的模型进行量化对比。
