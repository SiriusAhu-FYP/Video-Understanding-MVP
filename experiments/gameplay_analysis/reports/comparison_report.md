# 游戏画面分析专项实验 — 模型对比报告

**生成时间**: 2026-03-14 15:30  
**实验范围**: 6 款游戏截图 × 4 项任务 × 3 次重复 = 每模型 72 次推理  

| 模型 | 类型 | 参数量 |
|------|------|--------|
| Qwen/Qwen3-VL-2B-Instruct | 视觉语言模型 (VLM) | 2B |
| Qwen/Qwen3.5-0.8B | 纯文本语言模型 | 0.8B |

---

## 1. 性能指标对比

### 1.1 全局总览

| 指标 | Qwen3-VL-2B-Instruct | Qwen3.5-0.8B | 差异 |
|------|----------------------:|-------------:|------|
| 平均 TTFT (s) | **2.109** | 2.126 | 2B 略快 0.8% |
| 平均吞吐量 (tok/s) | 138.7 | **262.4** | 0.8B 快 **89%** |
| 平均总耗时 (s) | 4.747 | **3.887** | 0.8B 快 18% |
| 平均输出 Tokens | 371 | 470 | 0.8B 多 27% |
| VRAM 占用 (MB) | **11,276** | 11,741 | 2B 少 4% |
| 成功率 | 72/72 (100%) | 72/72 (100%) | 持平 |

**小结**: TTFT（首 token 延迟）两者几乎一致（~2.1s），主要受 prefill 阶段图片处理开销影响。吞吐量方面 0.8B 凭借更小的参数量实现了接近两倍的 decode 速度，总耗时也因此更短。

## 2. 输出质量对比

### 2.1 UI 元素检测 (Bounding Box)

#### 格式合规性
两个模型均能按照要求输出 JSON 格式的 bbox 数据。由于 `max_tokens=1024` 限制，大多数输出在列举到第 25-31 个元素时被截断，JSON 数组不完整。修复后的解析器能正确处理这种截断。

#### 检测质量

| 游戏 | 2B-Instruct | 0.8B |
|------|-------------|------|
| 明日方舟 | 31 个元素，但标签高度重复（大量 "敌人"），坐标呈均匀网格分布，疑似模式化输出而非精确检测 | 31 个元素，标签更多样（"设置按钮"、"能量条"、"等级"），但同样存在标签重复 |
| 赛博朋克 | 能识别 HUD 元素，坐标合理 | 元素数量类似，坐标分布不同 |
| 原神 | 仅 2 个元素，检测数量偏少 | 元素更多，但坐标准确性存疑（非 VLM 模型） |
| 我的世界 | 25 个元素 | 29 个元素 |
| P5R | 30 个元素 | 类似数量 |
| 马里奥 | 6 个元素 | 类似数量 |

**关键发现**: 2B 作为真正的 VLM，其 bbox 坐标基于实际视觉感知；0.8B 作为纯文本模型，bbox 坐标是基于文本理解的"猜测"。实际标注图中，2B 的框选位置与游戏 UI 的重合度更高。

### 2.2 画面描述 (scene_desc)

#### 明日方舟

| 模型 | 关键描述 | 准确性 |
|------|---------|--------|
| **2B** | "俯视角战斗画面"、"灰色和米色的方格瓷砖"、"敌方剩余数量为 46/59"、"单位限制为 2" | ✅ 准确读取 UI 数值，场景描述贴合实际 |
| **0.8B** | "左侧是一位白发女性角色，身穿红色战斗服"、"银发角色，装备着机械装置" | ⚠️ 角色外观描述为编造（无法真正看到图片），但整体叙事框架合理 |

#### 赛博朋克2077

| 模型 | 关键描述 | 准确性 |
|------|---------|--------|
| **2B** | "驾驶黑色高性能跑车"、车身 "LIFETIME" 字样、"角色杰基（Jackie）" 对话 | ✅ 精确读取车身文字和字幕内容 |
| **0.8B** | "黑色跑车"、车尾 "LIMITED" 字样、"Jackie" 对话 | ⚠️ 车身文字有误 (LIMITED vs LIFETIME)，但整体描述框架正确 |

#### 超级马里奥兄弟

| 模型 | 关键描述 | 准确性 |
|------|---------|--------|
| **2B** | 正确识别标题画面、"SUPER MARIO BROS." 招牌、马里奥外观、Goomba 描述 | ✅ 场景元素准确 |
| **0.8B** | 同样识别标题、招牌文字、角色位置 | ⚠️ 描述合理但细节可能基于训练数据中的游戏知识而非实际视觉感知 |

### 2.3 玩家意图推测 (player_intent)

两个模型在这项任务上的表现差距缩小，因为意图推测更依赖游戏领域知识而非精确视觉理解：

- **2B**: 给出的建议更具体，能引用画面中的实际信息（如 "敌方数量 46/59"），建议有针对性
- **0.8B**: 建议偏通用（"部署更多干员"、"观察敌人生成路线"），但作为游戏建议仍然合理

### 2.4 情况评价 (situation_eval)

两个模型都能按要求分【情感支持】和【客观评价】两部分输出：

- **2B**: 情感表达自然，客观评价能引用画面中的具体元素（如 "右上角高处站位"、"2 个单位"）
- **0.8B**: 情感表达同样到位，客观评价中的"干员部署"分析框架合理，但提到 "4 名干员" 等具体数字可能与实际不符

## 3. 核心结论

### 3.1 速度 vs 质量权衡

```
                        速度优势               质量优势
                    ◄───────────────────────────────────►
                    
    0.8B ████████████████░░░░                            (吞吐量 262 tok/s)
    2B   █████████░░░░░░░░░░░████████████████████████████ (吞吐量 139 tok/s)
```

### 3.2 各任务适用性建议

| 任务 | 推荐模型 | 理由 |
|------|---------|------|
| **UI 元素检测 (bbox)** | **2B-Instruct** | bbox 坐标需要真实视觉感知，0.8B 的坐标是基于文本推理的猜测 |
| **画面描述** | **2B-Instruct** | 能准确读取 UI 文字、数值和场景细节，0.8B 存在编造风险 |
| **玩家意图推测** | 均可，**2B 略优** | 两者都具备游戏领域知识，但 2B 能结合实际画面给出更精准建议 |
| **情况评价** | 均可，**2B 略优** | 情感支持质量相当，但 2B 的客观评价更基于事实 |

### 3.3 对项目的建议

对于视频理解 MVP 项目（游戏陪玩 AI），**强烈建议使用 Qwen3-VL-2B-Instruct**：

1. **视觉理解是核心需求**: 所有 4 个任务都依赖对游戏画面的准确感知，这是纯文本模型无法提供的
2. **速度差异可接受**: 2B 的吞吐量 (139 tok/s) 在实时场景中仍足够快，单次推理 3-5s 的延迟在陪玩场景中可接受
3. **0.8B 的"幻觉"风险**: 纯文本模型在视觉任务中会编造细节，在玩家需要精确信息（如 UI 状态、血量、技能冷却）的场景中会造成误导
4. **bbox 准确性**: 如果后续需要做 UI 元素追踪/自动化操作，2B 的 bbox 输出更可靠
